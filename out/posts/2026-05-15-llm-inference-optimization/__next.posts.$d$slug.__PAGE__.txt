3:T606f,<p class="text-[var(--color-text-2)] leading-relaxed my-4">大型语言模型（LLM）的推理速度和成本，是 2025-2026 年所有 AI 应用团队最头疼的问题之一。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">GPT-4o 生成一个回复要等 10 秒，Claude 4 的 API 账单比服务器费用还高，DeepSeek-V3 跑起来显存不够用——这些问题背后不是"模型不够好"，而是<strong class='text-[var(--color-text)] font-semibold'>推理工程</strong>的水太深。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">本文从系统角度拆解 LLM 推理的核心瓶颈，以及 2026 年主流的优化手段。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">推理的两阶段：Prefill 与 Decode</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">理解 LLM 推理优化的第一步，是把生成过程拆成两个本质不同的阶段：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>Prefill 阶段（输入处理）</strong>：把用户输入的 prompt 一次性通过模型，得到第一个 token。这个阶段本质上是<strong class='text-[var(--color-text)] font-semibold'>并行计算</strong>，充分利用 GPU 的矩阵运算能力，速度较快。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>Decode 阶段（自回归生成）</strong>：模型逐个 token 生成，每次生成都要"看一眼"之前所有的 token（包括自己刚生成的）。这个阶段是<strong class='text-[var(--color-text)] font-semibold'>顺序执行</strong>，每个 token 依赖于前一个，所以也叫"自回归"瓶颈。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">Prompt: "解释量子计算的基本原理，并举例说明其在密码学中的应用"</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">↓</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">[Prefill]  →  一次矩阵运算，处理所有输入 token</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">↓</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">第一个 token: "量子"</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">↓</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">[Decode]   →  逐个生成，每个 token 依赖之前所有 token</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">"量子" → "计算" → "的" → "基本" → "原理" → ...</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>关键洞察</strong>：Prefill 阶段用 GPU 并行，计算效率高；Decode 阶段是序列生成，GPU 利用率往往只有 10-30%，大量时间花在等待和内存访问上。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">瓶颈一：KV Cache 的内存墙</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">Decode 阶段最核心的问题是 <strong class='text-[var(--color-text)] font-semibold'>KV Cache</strong>。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">每个 Transformer 层都有一个 Key Cache 和 Value Cache，用来存储之前所有 token 的 Keys 和 Values。这样在计算注意力时，不需要重新计算已经处理过的 token。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">问题是：<strong class='text-[var(--color-text)] font-semibold'>随着上下文增长，KV Cache 线性膨胀</strong>。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">以 Llama 3 70B 为例：</p><ul class='space-y-2 my-5 ml-4 list-disc'><li class="text-[var(--color-text-2)] leading-relaxed">每个 token 在每层产生的 KV 数据约：`2 * hidden_size * 2 * num_heads * head_dim`</li><li class="text-[var(--color-text-2)] leading-relaxed">hidden_size=8192，num_heads=64，head_dim=128</li><li class="text-[var(--color-text-2)] leading-relaxed">每个 token 每层约：2 × 8192 × 2 × 64 × 128 / 8 = 32 MB？不对，重新算</li></ul><p class="text-[var(--color-text-2)] leading-relaxed my-4">实际上，每个 token 在 KV Cache 中占用的内存约为：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">参数总量 ≈ 70B</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">每个参数 float16 = 2 bytes</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">每个 token 的 KV Cache = 参数总量 / seq_len ≈ 140GB / 8192 ≈ 17MB/token/layer</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">70B 模型有 80 层 → 每个 token 全量 KV Cache ≈ 1.36 GB</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">这是<strong class='text-[var(--color-text)] font-semibold'>每个 token</strong> 的开销！所以 2048 个 token 的上下文，KV Cache 就能占用几 GB 到几十 GB 不等。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>Memory Bandwidth Wall（内存带宽墙）</strong>：Decode 阶段每个 token 生成时，都需要从显存读取完整的历史 KV Cache 数据。随着序列变长，内存带宽很快成为瓶颈，即使 GPU 计算单元还有大量空余。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">优化手段一：PagedAttention 与 vLLM</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">2023 年 vLLM 提出的 <strong class='text-[var(--color-text)] font-semibold'>PagedAttention</strong> 是近年来最重要的推理优化之一。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">核心思想：<strong class='text-[var(--color-text)] font-semibold'>把 KV Cache 当作"分页内存"来管理，而不是预先整块分配</strong>。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">传统方法的问题是：KV Cache 必须预先分配一个最大长度的连续显存（比如 4096 tokens），即使实际只用到 100 tokens，剩下 3996 个位置也被锁定了。这导致显存碎片化、利用率低。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># 传统方式的内存分配</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">KV Cache 预分配: [0.....0........0.....0]  # 4096 个位置，大部分空闲</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># PagedAttention 方式</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">KV Cache 分页管理: [Page 1][Page 2][Page 3]...  # 按需分配，动态拼接</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">[使用中]  [使用中]  [空闲]</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">vLLM 通过把 KV Cache 分成固定大小的"页"（类似操作系统内存分页），实现：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">1. <strong class='text-[var(--color-text)] font-semibold'>动态分配</strong>：需要多少分配多少，避免预分配的空间浪费</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">2. <strong class='text-[var(--color-text)] font-semibold'>共享显存</strong>：不同请求可以共享相同的前缀 KV Cache（用在 speculative decoding 或多请求复用场景）</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">3. <strong class='text-[var(--color-text)] font-semibold'>连续 Batching</strong>：更高效的请求调度</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">实测效果：vLLM 的吞吐量比 Hugging Face 默认实现高 <strong class='text-[var(--color-text)] font-semibold'>2-10 倍</strong>，在长上下文场景下提升尤为明显。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)]'>`</code>python</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># 传统 Static Batching</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># 所有请求同时开始，同时结束，短请求等待长请求</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># Continuous Batching</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">while running_requests:</p><p
