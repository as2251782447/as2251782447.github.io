10:T58b6,<h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">前言</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">传统可观测性靠的是应用层埋点——在你代码里插 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>traceSpan</code>、<code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>metrics.Inc()</code>，然后等着数据上报。这套东西在微服务少的时候挺好使，但上了 Kubernetes 几百个 Pod 之后，问题就来了：<strong class='text-[var(--color-text)] font-semibold'>盲区太多、性能损耗、升级改动大</strong>。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">eBPF 换了个思路：<strong class='text-[var(--color-text)] font-semibold'>不让应用主动上报，让内核帮你看</strong>。不需要改一行代码，就能拿到网络延迟、文件系统 I/O、内存分配、系统调用分布——全部在 kernel space 完成，对应用零侵入。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">今天聊一下 eBPF 在云原生环境里的实际用法，包括如何用 Cilium 替换 kube-proxy、怎么用 eBPF 程序做网络流量分析、以及生产级部署的坑。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">---</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">1. 为什么传统埋点会瓶颈</h2><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">1.1 注入式 APM 的代价</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">传统方案（Jaeger、Datadog APM、OpenTelemetry）的工作原理是：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">1. SDK 注入到你的应用进程</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">2. 每个请求路过 SDK 时记录 span</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">3. 数据通过 HTTP/gRPC 发送到一个 collector</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">4. Collector 处理后存到后端</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">问题在哪？<strong class='text-[var(--color-text)] font-semibold'>每一个 span 都有开销</strong>。在你高频调用的路径上，这个开销会变成真实延迟。2024 年 AWS 的一个内部测试显示，在 Go 服务里启用完整链路追踪会增加 <strong class='text-[var(--color-text)] font-semibold'>8-15% 的 P99 延迟</strong>。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">另外，语言/框架多样性带来碎片化。你 Node.js、Python、Go 各一套 SDK，升级策略完全不同，有的地方漏埋了就是盲区。</p><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">1.2 Kubernetes 网络的可观测性真空</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">在 K8s 里，Pod 间的流量不经过物理网卡，而是走 veth pair → docker0/bridge → iptables/nftables → 路由。这个路径上，传统抓包工具（tcpdump）看不到加密流量，sidecar 模式（如 Istio envoy）注入了额外的 hop，而服务网格的自定义指标覆盖率也无法达到 100%。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">---</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">2. eBPF 工作原理快速入门</h2><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">2.1 什么是 eBPF</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">eBPF（extended Berkeley Packet Filter）是 Linux kernel 里的一个沙箱虚拟机。你写一段程序，kernel 验证它安全（不会死循环、不会越界访问），然后把它 attach 到某个内核 hook 上执行。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">关键特性：</p><ul class='space-y-2 my-5 ml-4 list-disc'><li class="text-[var(--color-text-2)] leading-relaxed">**内核沙箱验证**：不安全代码直接拒绝，不会上线</li><li class="text-[var(--color-text-2)] leading-relaxed">**内核直接执行**：不需要进用户态，少一次 context switch</li><li class="text-[var(--color-text-2)] leading-relaxed">**可动态 attach/detach**：不需要重启进程，不需要改代码</li></ul><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">2.2 程序类型和 Hook 点</h3><div class="flex gap-3 my-2 text-sm text-[var(--color-text-2)]"><span class="flex-1"> 程序类型 </span><span class='flex-1'> Hook 点 </span><span class='flex-1'> 用途 </span></div><div class="flex gap-3 my-2 text-sm text-[var(--color-text-2)]"><span class="flex-1"> `socket_filter` </span><span class='flex-1'> 协议栈入口 </span><span class='flex-1'> 抓包/流量镜像 </span></div><div class="flex gap-3 my-2 text-sm text-[var(--color-text-2)]"><span class="flex-1"> `kprobe` </span><span class='flex-1'> 任意内核函数 </span><span class='flex-1'> 系统调用追踪 </span></div><div class="flex gap-3 my-2 text-sm class="text-[var(--color-text-2)] leading-relaxed my-4">#!/usr/bin/env python3</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">from bcc import BPF</p><p class="text-[var(--color-text-2)] u32);</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">int count_tcp_accept(struct pt_regs *ctx) {</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">u32 sip =
