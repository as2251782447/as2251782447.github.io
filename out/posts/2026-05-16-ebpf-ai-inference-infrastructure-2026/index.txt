10:T5d6f,<p class="text-[var(--color-text-2)] leading-relaxed my-4">大多数人接触 eBPF 是因为 <strong class='text-[var(--color-text)] font-semibold'>云原生可观测性</strong>：用它来抓包、追踪系统调用、分析网络流量。这套叙事在 2019–2024 年非常流行，工具链（Cilium、Tetragon、Pixie）基本都建立在这个框架上。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">但 2026 年的 eBPF 正在发生一次静默的范式转移。它正在从"观测工具"变成"AI 推理基础设施层"——一个直接运行在 Linux 内核中、以内核态权限实时感知 GPU 内存、CUDA 流和模型计算图的组件。这个转变的驱动力很简单：GPU 太贵了，延迟太关键了，在用户态做监控已经不够用了。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">为什么用户态监控不够用</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">先从一个问题出发：<strong class='text-[var(--color-text)] font-semibold'>你如何在 1ms 级别感知 GPU 显存使用量的变化？</strong></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">用 nvidia-smi 轮询？最乐观的轮询间隔是 100ms，且本身会触发 GPU 上下文切换，引入 0.5–2ms 的额外延迟。在一个推理请求希望在 50ms 内完成全部计算的场景里，nvidia-smi 的监控开销已经不可忽视。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">用 CUDA Profiling API（cuBLAS、cuDNN 的回调机制）？这需要修改模型代码，或者依赖特定的 CUDA 版本。实际生产环境的模型通常不集成这些 API，而且 profiling 数据会通过 PCIe 传回主机端，再次引入延迟。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>eBPF 的解题思路是：在 GPU 驱动层直接插桩，内核态接收数据，不需要任何模型侧改动。</strong></p><h2 class="text-2xl font-black <strong class='text-[var(--color-text)] font-semibold'>NVML（NVIDIA Management Library）</strong> 提供设备状态查询。NVML 本身通过 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>nvidiafs</code> 或直接映射的 PCIe BAR 区域与 GPU 通信。内核模块 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>nvidia.ko</code>（以及 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>nvidia-uvm.ko</code>）负责这个通信。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">从 Linux 6.6 开始，NVIDIA 驱动开始导出 <strong class='text-[var(--color-text)] font-semibold'>per-process GPU 内存配额信息</strong> 到 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border font-mono border border-[var(--color-border)]'>nvidia-smi</code> 用的，但 eBPF 程序可以直接读取。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">更重要的是，2026 年的 eBPF 已经支持 <strong class='text-[var(--color-text)] font-semibold'>直接 attach 到 NVIDIA 驱动的 tracepoint</strong>：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code>bash</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># 查看当前 NVIDIA 驱动暴露的 tracepoints</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">$ sudo ls /sys/kernel/debug/tracing/events/nvidia_gpu/</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">nvidia_pmu__gpu_activity     nvidia_pmu__mem_usage</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">nvidia_pmu__compute_inst     nvidia_pmu__memory_controller</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">这意味着你可以在<strong class='text-[var(--color-text)] font-semibold'>每一次 GPU 内存分配/释放发生时</strong>，在 kernel my-4">bpf_map_update_elem(&gpu_mem_peaks, &pid, &new_peak, BPF_ANY);</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">} else {</p><p class="text-[var(--color-text-2)] leading-relaxed class="text-[var(--color-text-2)] leading-relaxed my-4">u64 bytes;</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">u64 timestamp_ns;</p><p class="text-[var(--color-text-2)] class="text-[var(--color-text-2)] leading-relaxed my-4">sizeof(struct gpu_mem_event), 0);</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">if (!event)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">return 0;</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">event->pid = pid;</p><p class="text-[var(--color-text-2)] 在推理调度中的应用</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">更激进的用法是<strong class='text-[var(--color-text)] font-semibold'>把 eBPF 作为推理调度的决策源</strong>。</p><p font-semibold'>动态 batch 路由器</strong>：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">请求进来</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">→ eBPF 查询当前各模型的 GPU 内存占用（从 ring buffer 聚合）</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">→ 如果最小模型的剩余显存足够 → 分发到最小模型（最低延迟）</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">→ 如果不够但大模型有足够空间 → 分发到大模型</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">→ 如果都不够 → 进入队列等待</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">这个逻辑可以实现在一个 Linux tc（traffic control）ebpf 程序里，直接在网络层做请求路由，不需要任何外部负载均衡器。内核态决策，零网络跳数。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">2026 年已经有团队在生产环境中验证了类似方案：据公开资料，ByteDance 在 2025 年底的开源项目 [Merak](https://github.com/bytedance/merak) 就采用了类似的思路，用 eBPF 做多模型推理服务的动态调度。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">eBPF 可编程内核网络 + GPU 的联合监控</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">还有一个更前沿的方向：<strong class='text-[var(--color-text)] font-semibold'>把 CPU 侧的网络 trace 和 GPU 侧的推理 trace 关联起来</strong>，形成端到端的请求延迟剖析。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">典型推理请求的完整路径：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">HTTP 请求进入 → Nginx 接收 → upstream 分发 → 模型推理 → HTTP 响应</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">↑                ↑              ↑              ↑</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">net trace      eBPF net      eBPF GPU trace   net trace</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">用 eBPF 的 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>sock_ops</code> 或 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>raw_tracepoint/sock_sendmsg</code> 可以追踪请求在各网络节点的到达和离开时间；用 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>nvidia_pmu</code> tracepoint 追踪 GPU 计算阶段。两组数据打上相同的时间戳和服务端 PID，就能还原出请求在整个链路上各阶段的时间分布。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">这种端到端剖析在没有 eBPF 的时代需要每个组件单独插桩（Pinpoint、SkyWalking、各语言埋点），且无法拿到 GPU 侧的真实数据。用 eBPF 可以做到<strong class='text-[var(--color-text)] font-semibold'>一次 probe，覆盖全链路</strong>。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">当前限制和工程挑战</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">吹完了也得说实话。当前 eBPF 在 AI 基础设施方向有几个工程挑战：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>1. NVIDIA 驱动版本耦合</strong></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">上面提到的 tracepoints（<code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>nvidia_pmu::*</code>）只在 <strong class='text-[var(--color-text)] font-semibold'>535.x 以上的驱动版本</strong> 中默认开启。旧版本或开源 nouveau 驱动没有这些接口。如果生产环境跑的是 AWS Graviton 或其他非 NVIDIA GPU，这套方案完全不可用。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>2. 内核版本要求</strong></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">BPF ring buffer 在 Linux 5.8 引入，BPF CO-RE（Compile Once – Run Everywhere）在 5.10 稳定。生产服务器的内核通常保守，很多还在跑 5.4 / 5.8。如果要推广，需要在内核升级和稳定性之间做权衡。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>3. 权限和安全</strong></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">eBPF 程序需要 root 或 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>CAP_SYS_ADMIN</code> 能力。在多租户环境中，这通常是一个安全顾虑。用 <strong class='text-[var(--color-text)] font-semibold'>bpf() system call with restricted capabilities</strong> 和 seccomp 过滤可以缓解，但不是开箱即用。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>4. 数据量</strong></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">在高频推理场景下，<code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>nvidia_pmu::mem_usage</code> 可能每秒触发数万次。如果每个事件都发送到 ring buffer，用户态处理可能成为瓶颈。实践中需要：</p><ul class='space-y-2 my-5 ml-4 list-disc'><li class="text-[var(--color-text-2)] leading-relaxed">**采样**：不是每个事件都上报，而是统计窗口内的峰值</li><li class="text-[var(--color-text-2)] leading-relaxed">**内核聚合**：在 eBPF 程序内部计算 min/max/count，输出聚合结果而非原始事件</li><li class="text-[var(--color-text-2)] leading-relaxed">**共享内存传递**：用 `BPF_MAP_TYPE_PERCPU_HASH` 做本地聚合，然后批量推送给用户态消费者</li></ul><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">总结</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">eBPF 正在从"云原生可观测性工具"演化为"AI 推理基础设施层"。它的核心价值在于：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">1. <strong class='text-[var(--color-text)] font-semibold'>内核态数据获取</strong>：绕过用户态，直接从驱动层拿 GPU 指标</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">2. <strong class='text-[var(--color-text)] font-semibold'>零侵入监控</strong>：不需要修改模型代码，不需要 CUDA Profiling API</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">3. <strong class='text-[var(--color-text)] font-semibold'>实时决策</strong>：ring buffer 数据可以直接驱动调度逻辑</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">4. <strong class='text-[var(--color-text)] font-semibold'>端到端可关联</strong>：联合 CPU 网络 trace 和 GPU 计算 trace，实现真正的一次性全链路剖析</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">限制也很明确：驱动版本耦合、内核版本要求、权限问题。但这些都是工程问题，不是架构问题。在 GPU 成本压力持续增加、推理延迟要求持续降低的背景下，eBPF 作为 AI 基础设施层的故事才刚刚开始。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">下一步值得关注的方向是 [Merak](https://github.com/bytedance/merak) 和 [rBPF](https://github.com/facebookexperimental/rbpf) 在生产推理集群中的实践，以及 Linux 社区是否会推进标准的 GPU tracepoint 接口——如果 NVIDIA 和 AMD 能就统一的 GPU PMU 接口达成共识，eBPF 在 AI
