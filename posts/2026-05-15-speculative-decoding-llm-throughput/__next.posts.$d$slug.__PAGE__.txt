3:T8813,<p class="text-[var(--color-text-2)] leading-relaxed my-4">做 LLM 推理服务的工程师都知道，自回归解码（Autoregressive Decoding）是延迟的罪魁祸首：每个 token 依赖前一个 token 生成，串行化严重，GPU 利用率惨不忍睹。H100 的算力利用率在 LLM 解码时往往只有 <strong class='text-[var(--color-text)] font-semibold'>30-50%</strong>，大量时间花在"等上一个 token 生成"上。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>推测解码（Speculative Decoding）</strong> 是 2022 年底 Google 提出的一种技术，核心思路：用一个小模型"猜"多个 token，再用大模型并行验证，一把解决串行瓶颈。2026 年的今天，这套技术已经在生产环境大规模落地，效果经过验证。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">一、为什么自回归解码是性能瓶颈</h2><p class="text-[var(--color-text-2)] leading-relaxed my-4">在说推测解码之前，先弄清楚问题在哪。</p><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">1.1 自回归解码的串行困境</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">Transformer 的推理分两个阶段：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>Prefill 阶段</strong>：一次性处理完整 prompt，KV Cache 构建，全部 token 并行计算。这一步很快。</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><strong class='text-[var(--color-text)] font-semibold'>Decode 阶段</strong>：逐个生成 token。每次生成时，模型要用最新生成的 token + 之前所有 token 的 KV Cache 做计算。由于每个 token 依赖前一个的输出，<strong class='text-[var(--color-text)] font-semibold'>这一步天然串行</strong>：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">Token 1 → Token 2 → Token 3 → Token 4 → Token 5 → ...</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">↓         ↓         ↓         ↓         ↓</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">compute   wait      wait      wait      wait</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">一个 70B 参数的模型，Decode 阶段每个 token 生成需要约 <strong class='text-[var(--color-text)] font-semibold'>50-100ms</strong>（A100）。生成 200 个 token 的回复就需要 <strong class='text-[var(--color-text)] font-semibold'>10-20 秒</strong>。而这中间，GPU 大量时间在等待——不是算力不够，是数据依赖导致无法并行。</p><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">1.2 算力利用率低下的本质</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">GPU 是并行计算设备，最怕的是<strong class='text-[var(--color-text)] font-semibold'>数据依赖</strong>。Decode 阶段每个 token 的计算都依赖前一个 token 的结果，导致：</p><ul class='space-y-2 my-5 ml-4 list-disc'><li class="text-[var(--color-text-2)] leading-relaxed">KV Cache 需要反复读写（memory bound）</li><li class="text-[var(--color-text-2)] leading-relaxed">GPU SM（Streaming Multiprocessor）利用率低</li><li class="text-[var(--color-text-2)] leading-relaxed">批处理（batch）在 decode 时几乎无效（batch 中不同序列长度不同，无法完全对齐）</li></ul><p class="text-[var(--color-text-2)] leading-relaxed my-4">这就是为什么即使在高端 H100 上，LLM Decode 的吞吐量也比峰值低得多。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">二、推测解码的原理</h2><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">2.1 核心思想：用小模型"猜"，大模型"验"</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">推测解码的基本框架：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">1. <strong class='text-[var(--color-text)] font-semibold'>小模型（Draft Model）</strong>：轻量级模型，推理快，生成 K 个候选 token</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">2. <strong class='text-[var(--color-text)] font-semibold'>大模型（Target Model）</strong>：主力模型，一次性对 K 个 token 做并行验证</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">3. <strong class='text-[var(--color-text)] font-semibold'>接受/拒绝</strong>：根据大模型的概率分布，决定保留哪些 token，拒绝哪些</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">传统（串行）：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">大模型 → token1 → token2 → token3 → token4 → token5</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">推测解码（并行验证）：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">小模型 → token1* → token2* → token3* → token4*  (快速猜测)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">大模型 → [验证 token1* token2* token3* token4*]  (并行验证)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">接受    ✓         ✓        ✗（重新生成） skip</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">如果小模型猜对了，大模型只需要做一次并行验证，节省了串行等待的时间。如果猜错了，大模型会纠正，流程继续。</p><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">2.2 验证算法：基于接受概率</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">关键问题：如何判断小模型猜的 token 是否"够好"？</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">最常见的方案是<strong class='text-[var(--color-text)] font-semibold'>基于概率比值的拒绝采样</strong>（类似 GSmart):</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code>python</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">import torch</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">import torch.nn.functional as F</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">def speculative_verify(</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">draft_tokens: torch.Tensor,       # [batch, k] 小模型猜的 K 个 token</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">draft_probs: torch.Tensor,         # [batch, k, vocab] 小模型的概率分布</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">target_logits: torch.Tensor,       # [batch, k+1, vocab] 大模型的输出 logits</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">temperature: float = 1.0,</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">gamma: int = 4,                    # 小模型每次猜的 token 数</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">eta: float = 0.3,                  # 接受阈值</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">) -> tuple[list[int], int, int]:</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">"""</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">返回: (接受的 token 列表, 小模型生成数, 实际接受数)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">"""</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">batch_size = draft_tokens.shape[0]</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">target_probs = F.softmax(target_logits[:, :-1] / temperature, dim=-1)  # [batch, k, vocab]</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">accepted = []</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">total_draft = 0</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">total_accepted = 0</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">for seq_idx in range(batch_size):</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">seq_accepted = []</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">for i in range(gamma):</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">t = draft_tokens[seq_idx, i]</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">p_draft = draft_probs[seq_idx, i, t]</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">p_target = target_probs[seq_idx, i, t]</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># 接受概率 = min(1, p_target / p_draft)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">acceptance_ratio = min(1.0, (p_target / (p_draft + 1e-10)).item())</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">if random.random() < acceptance_ratio:</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">seq_accepted.append(t.item())</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">total_accepted += 1</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">else:</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"># 拒绝：这里本应采样一个纠正 token，简化处理直接跳过</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">break</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">total_draft += gamma</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">accepted.append(seq_accepted)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">return accepted, total_draft, total_accepted</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">这里的核心判断是：如果大模型认为这个 token 的概率 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>p_target</code> 显著高于小模型的 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>p_draft</code>，则接受。如果 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>p_target</code> 接近 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>p_draft</code>，则以 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>p_target/p_draft</code> 的概率接受。如果 <code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>p_target << p_draft</code>，则几乎一定拒绝。</p><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">2.3 树状验证：更激进的并行</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">基础的 gamma=4 方案，每一步还是串行地一个 token 一个 token 地验证。进阶的<strong class='text-[var(--color-text)] font-semibold'>树状推测解码（Tree Speculative Decoding）</strong> 把这个过程变成一颗树：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">小模型猜：token1 → token2 → token3 → token4</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">└── token2a → token2b（分支）</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">└── token3a（更深的分支）</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">大模型并行验证整棵树，而不是线性链</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)] leading-relaxed my-4">这样可以用一次 forward pass 验证更多候选路径。但树的结构设计、接受率与深度的权衡，都是工程上需要精心调优的。</p><h2 class="text-2xl font-black text-[var(--color-text)] mt-12 mb-4 tracking-tight border-b border-[var(--color-border)] pb-3">三、生产级实现：Hugging Face Transformers 的内置支持</h2><p class="text-[var(--color-text-2)] leading-relaxed AutoModelForCausalLM.from_pretrained(</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">draft_model_id,</p><p class="text-[var(--color-text-2)] leading-relaxed "解释一下量子纠缠的基本原理："</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">inputs = tokenizer(prompt, return_tensors="cuda")</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">outputs = target_model.generate(</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">**inputs,</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">**generation_config</p><p leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><h3 class="text-base font-bold text-[var(--color-text)] mt-8 mb-3">3.2 vLLM 的推测解码实现</h3><p class="text-[var(--color-text-2)] leading-relaxed my-4">vLLM 是目前最流行的高吞吐量推理框架。它的推测解码实现做了大量工程优化：</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code>python</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">from vllm import LLM, SamplingParams</p><p class="text-[var(--color-text-2)] leading-relaxed # gamma 值</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">speculative_eta=0.3,       # 接受阈值</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4">outputs = llm.generate(["解释量子纠缠"], sampling_params)</p><p class="text-[var(--color-text-2)] leading-relaxed my-4"><code class='bg-[var(--color-surface)] text-[var(--color-accent)] px-1.5 py-0.5 rounded text-sm font-mono border border-[var(--color-border)]'>`</code></p><p class="text-[var(--color-text-2)]
